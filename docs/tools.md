# TASO Tool Reference

Tools are modular, discoverable Python classes that implement a standard
interface.  The `ToolRegistry` auto-discovers all `BaseTool` subclasses in
`tools/` at startup.

---

## Built-in Tools

| Tool | Class | Description |
|---|---|---|
| `repo_analyzer` | RepoAnalyzerTool | Clone + analyse a Git repo |
| `dependency_scanner` | DependencyScannerTool | Scan requirements.txt for CVEs |
| `web_crawler` | WebCrawlerTool | Fetch and parse web pages / CVE feeds |
| `system_monitor` | SystemMonitorTool | Host CPU / memory / disk snapshot |
| `git_manager` | GitManagerTool | Git operations (clone, commit, push) |
| `log_analyzer` | LogAnalyzerTool | Analyse log files for patterns |
| `log_monitor` | LogMonitorTool | Live scan logs for error counts |

## Tool Interface

Every tool subclasses `BaseTool` and exposes:

```python
class MyTool(BaseTool):
    name        = "my_tool"           # unique snake_case identifier
    description = "Does X."          # used by agents to select tools
    schema      = ToolSchema({        # input field definitions
        "target": {"type": "str", "required": True},
        "limit":  {"type": "int", "required": False, "default": 10},
    })

    async def execute(self, target: str, limit: int = 10) -> Dict:
        # Do the work. Return raw result dict (not wrapped).
        return {"results": [...], "count": N}
```

`BaseTool.run()` wraps `execute()` automatically:
```python
{"success": True, "result": <execute() return>, "error": None}
```

## Dynamic Tool Generation

Agents can generate new tools at runtime:

### Via Telegram
```
/create_tool A tool that fetches the latest CVEs for a given keyword from NVD
```

### Via Python
```python
from agents.developer_agent import DeveloperAgent
from agents.message_bus import bus

dev = DeveloperAgent(bus)
# Sends: developer.request { action: "generate_tool", task: "..." }
```

### Pipeline
1. `DynamicToolGenerator.generate(description)` — LLM writes `run_tool(input_data)`
2. `sandbox_tester.sandbox_test_tool(code)` — runs in subprocess
3. If passed: `ToolRegistry.register_dynamic(name, code, ...)`
4. All agents can immediately call the new tool via `registry.get(name)`
5. `AuditLog.record(action="tool_created")` + VersionHistoryDB entry

## Tool Registry API

```python
from tools.base_tool import registry

# List all tools
tools = registry.list_tools()          # → [{"name": ..., "description": ...}, ...]

# Get a tool
tool = registry.get("port_scanner")   # → PortScannerTool instance

# Run a tool
result = await registry.get("disk_usage").run(paths=["/", "/tmp"])

# Check if tool exists
if "my_tool" in registry:
    ...

# Register a dynamic tool
registry.register_dynamic(
    name="my_tool",
    code="...",
    description="Does X",
    input_schema={"target": {"type": "str", "required": True}},
    output_schema={},
    tags=["network"],
    version="1.0.0",
)
```

## Sandboxed Tool Execution

All code generated by `DynamicToolGenerator` runs through
`tools/sandbox_tester.py` before registration:

```python
from tools.sandbox_tester import sandbox_test_tool

passed, output = await sandbox_test_tool(code)
```

The sandbox:
- Wraps generated code in a test harness
- Runs inside a subprocess with 10-second timeout
- Captures stdout/stderr
- Returns `(True, output)` on success, `(False, error)` on failure
